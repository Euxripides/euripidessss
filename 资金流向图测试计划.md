# 资金流向图测试计划

> 文档版本：v2.0  
> 编制日期：2026-05-27  
> 适用项目：`E:\codex\etl` 资金数据智能分析平台  
> 重点目标：保证资金流向图中每一条边、每一个节点、每一笔金额、每一个方向、每一条明细均可回溯并能与原始流水逐条核对。  

## 1. 测试目标

1. 验证资金流向图从 CSV 手工导入、数据库导入、字段映射、数据筛选、建图、边详情、主体详情、导出到 UI 展示的全链路准确性。
2. 验证所有金额统计、笔数统计、方向归一化、节点关系、边关系、时间顺序、账户归属、字段映射、去重、筛选和聚合逻辑均能与原始流水逐条核对。
3. 验证同名账户、同账号不同户名、同户名不同账号、空值、脏数据、重复流水、冲正、退款、手续费、收入、支出、转账、红包、提现、充值等数据形态不会导致错边、错向、错金额或漏明细。
4. 验证单账户、多账户、多层级、多路径、循环转账、拆分转账、汇总转账、间接流向、亲属账户、嫌疑人账户、被害人账户、第三方账户等调查场景准确呈现。
5. 验证百万级、千万级、上亿级场景下的性能、内存、分页、懒加载、聚合计算、并发计算和数据库索引策略。
6. 建立缺陷判定、复现、修复、回归和验收闭环，任何数据准确性缺陷必须有最小复现数据、自动化测试和真实数据回归。

### 4.2 追溯账本

每次执行数据准确性测试必须生成“追溯账本”。追溯账本可以是 CSV、临时数据库表或测试日志，但必须至少包含以下字段：

| 字段 | 说明 | 生成规则 |
|---|---|---|
| `source_type` | 数据来源 | `csv`、`pgsql`、`manual_upload`、`db_import`、`synthetic` |
| `source_ref` | 来源位置 | CSV 绝对路径或 `mz.ls_0709.交易明细信息` |
| `source_row_no` | 原始行号 | CSV 从 2 开始；PG 使用稳定排序后的 `row_number()` 或主键 |
| `row_hash` | 原始行哈希 | 对核心字段拼接后计算 SHA256 |
| `transaction_id` | 流水标识 | 优先 `交易流水号`，缺失时用复合键 |
| `raw_source_account` | 原始本方账号 | 来自 `交易卡号` 或 `交易账号` |
| `raw_source_name` | 原始本方户名 | 来自 `交易方户名` |
| `raw_target_account` | 原始对手账号 | 来自 `交易对手账卡号` |
| `raw_target_name` | 原始对手户名 | 来自 `对手户名` |
| `raw_direction` | 原始收付标志 | 不允许覆盖 |
| `normalized_direction` | 归一化方向 | `进`、`出`、空、未知 |
| `raw_amount` | 原始金额 | 保留文本或 numeric 原值 |
| `normalized_amount` | 参与计算金额 | 统一为两位小数 |
| `raw_time` | 原始交易时间 | 保留原值 |
| `normalized_time` | 归一化时间 | 后端参与排序和筛选的时间 |
| `edge_source` | 建图源节点 | 方向为 `出` 时本方，方向为 `进` 时对手 |
| `edge_target` | 建图目标节点 | 方向为 `出` 时对手，方向为 `进` 时本方 |
| `edge_id` | 边 ID | 与后端返回一致 |
| `filter_id` | 筛选批次 | 每一组筛选条件一个独立编号 |
| `included_by_filter` | 是否命中筛选 | `true` 或 `false` |

通过标准：

1. 任意边 `tx_count = 追溯账本中 edge_source + edge_target 命中行数`。
2. 任意边 `amount = 追溯账本中 normalized_amount 求和`，误差不得超过 `0.005` 元。
3. 任意节点 `amount_in = 所有入边金额合计`，`amount_out = 所有出边金额合计`。
4. 主体详情和边详情里的每一行必须能用 `row_hash` 或 `transaction_id` 回查原始流水。

### 4.3 PG 基准 SQL

```sql
-- 行数与金额总览
SELECT COUNT(*) AS row_count, SUM("交易金额") AS amount_sum
FROM ls_0709."交易明细信息";
```

```sql
-- 原始方向分布
SELECT "收付标志", COUNT(*) AS cnt, SUM("交易金额") AS amount
FROM ls_0709."交易明细信息"
GROUP BY "收付标志"
ORDER BY cnt DESC;
```

```sql
-- 建图方向基准。出：本方 -> 对手；进：对手 -> 本方。
WITH base AS (
  SELECT
    COALESCE(NULLIF("交易账号", ''), NULLIF("交易卡号", ''), NULLIF("交易方户名", ''), '本方未知') AS own_id,
    COALESCE(NULLIF("交易对手账卡号", ''), NULLIF("对手户名", ''), '对手未知') AS counter_id,
    "收付标志" AS direction,
    "交易金额"::numeric(38, 2) AS amount,
    "交易时间" AS trade_time,
    "交易流水号" AS serial_no
  FROM ls_0709."交易明细信息"
),
edges AS (
  SELECT
    CASE WHEN direction = '出' THEN own_id WHEN direction = '进' THEN counter_id END AS source_id,
    CASE WHEN direction = '出' THEN counter_id WHEN direction = '进' THEN own_id END AS target_id,
    amount,
    trade_time,
    serial_no
  FROM base
  WHERE direction IN ('进', '出')
)
SELECT source_id, target_id, COUNT(*) AS tx_count, SUM(amount) AS amount,
       MIN(trade_time) AS first_time, MAX(trade_time) AS last_time
FROM edges
WHERE source_id IS NOT NULL AND target_id IS NOT NULL AND source_id <> target_id
GROUP BY source_id, target_id
ORDER BY amount DESC;
```

```sql
-- 节点统计基准。
WITH edge_base AS (
  SELECT
    CASE WHEN "收付标志" = '出' THEN COALESCE(NULLIF("交易账号", ''), NULLIF("交易卡号", ''), NULLIF("交易方户名", ''), '本方未知')
         WHEN "收付标志" = '进' THEN COALESCE(NULLIF("交易对手账卡号", ''), NULLIF("对手户名", ''), '对手未知') END AS source_id,
    CASE WHEN "收付标志" = '出' THEN COALESCE(NULLIF("交易对手账卡号", ''), NULLIF("对手户名", ''), '对手未知')
         WHEN "收付标志" = '进' THEN COALESCE(NULLIF("交易账号", ''), NULLIF("交易卡号", ''), NULLIF("交易方户名", ''), '本方未知') END AS target_id,
    "交易金额"::numeric(38,2) AS amount
  FROM ls_0709."交易明细信息"
  WHERE "收付标志" IN ('进', '出')
)
SELECT node_id,
       SUM(amount_in) AS amount_in,
       SUM(amount_out) AS amount_out,
       SUM(in_count) AS in_count,
       SUM(out_count) AS out_count
FROM (
  SELECT target_id AS node_id, SUM(amount) AS amount_in, 0::numeric AS amount_out, COUNT(*) AS in_count, 0 AS out_count
  FROM edge_base
  WHERE source_id <> target_id
  GROUP BY target_id
  UNION ALL
  SELECT source_id AS node_id, 0::numeric AS amount_in, SUM(amount) AS amount_out, 0 AS in_count, COUNT(*) AS out_count
  FROM edge_base
  WHERE source_id <> target_id
  GROUP BY source_id
) t
GROUP BY node_id;
```

## 5. 总体执行流程

1. 建立测试基线：运行 Go 后端测试、前端构建、健康检查，确认当前版本可运行。
2. 采集原始基准：对 CSV 和 PG 分别计算行数、方向分布、金额合计、时间范围、Top 账号、Top 关系。
3. 执行手工导入路径：上传 CSV，确认字段识别、映射、导入预览、建图、筛选、详情、导出。
4. 执行数据库导入路径：配置 PG 连接，预览表，确认字段映射，启动导入任务，建图并与 CSV 同源结果核对。
5. 执行专项数据：同名账户、同账号不同户名、空值、脏数据、退款、冲正、手续费、红包、循环转账、拆分转账等。
6. 执行筛选联动：逐组切换来源、目标、方向、时间、金额、标签、明细筛选，验证图、统计、明细、导出同步变化。
7. 执行大数据和并发测试：百万级、千万级、上亿级分层验证，记录耗时、内存、索引命中和错误率。
8. 执行 UI 和导出测试：ReactFlow 画布、边详情、主体详情、图层、数据穿透、导出文件可用性。
9. 执行安全测试：路径穿越、非法 SQL、密码泄露、危险文件、跨 session 访问。
10. 汇总缺陷：任何 P0/P1 缺陷必须最小复现、补测试、修复、真实 CSV/PG 回归后关闭。

## 9. 缺陷判定标准

### 9.1 P0 致命缺陷

出现以下任一情况必须停止发布并立即修复：

1. 任意金额与原始流水差异大于或等于 `0.01` 元。
2. 任意方向反转错误，例如原始出账被展示为入账。
3. 筛选后行数、金额、边集合与 SQL 或追溯账本不一致。
4. 边详情 `tx_count`、金额与边上展示不一致。
5. 主体详情流入/流出与节点关联边不一致。
6. 数据库导入与手工导入同源数据结果不一致，且无法解释。
7. 任一交易在无合理规则说明的情况下丢失或重复计算。
8. API 在核心建图、筛选、详情路径返回 500、panic 或空图但原始数据非空。

### 9.2 P1 严重缺陷

1. 字段映射错误导致主体名称、账号、身份证来源错误。
2. 筛选单调性失败，即增加筛选条件后结果反而增加。
3. 大文件导入、边详情分页、导出出现稳定失败。
4. 数据穿透、边详情、主体详情等关键交互不可用。
5. 密码、数据库连接串、服务器路径出现在 API 响应、导出文件或日志中。
6. 非 SELECT SQL 未被拦截，或路径穿越可读取本地文件。

### 9.3 P2 一般缺陷

1. 导出图像裁剪、格式不兼容，但数据导出仍准确。
2. UI 样式错位、按钮文本溢出、提示不清晰。
3. 非核心字段展示缺失，但不影响金额、方向、节点、边准确性。

### 9.4 P3 轻微缺陷

1. 文案、排序默认值、颜色、图标等轻微问题。
2. 对准确性无影响的日志和提示优化。

## 11. 缺陷修复闭环

任何 P0/P1 缺陷必须按以下流程关闭：

1. 记录失败场景：session_id、filter_id、source、target、edge_id、节点 ID、截图、API payload、响应、SQL 基准。
2. 抽取最小复现数据：从 CSV 或 PG 中导出能复现的最小行集，保留 `source_row_no` 和 `row_hash`。
3. 增加自动化测试：优先加入 `internal/api/handlers_test.go`、`internal/etl/flow_graph_test.go` 或专项测试文件。
4. 修复代码：只修改缺陷相关模块，不重构无关模块。
5. 运行回归：最小用例、真实 CSV、PG 审计、前端构建全部通过。
6. 更新交接：在 `docs/AI_HANDOFF.md` 和 `docs/CHANGELOG_AI.md` 记录修改文件、接口变化、数据库变化、验证命令、未完成事项。
