## 1. 测试概述

### 1.1 测试目标

1. **数据准确性**: 资金流向图中每一条边、每一个节点、每一个金额、每一个方向都必须可追溯到原始流水数据
2. **金额核对**: 所有统计金额必须能和原始数据逐条核对，不存在重复计算、漏算、错算
3. **方向核对**: 方向归一化（进/出）必须正确，借/贷/D/C/收入/支出等别名全部正确映射
4. **节点关系**: 每个节点的流入/流出统计与关联边的金额合计严格一致
5. **边关系**: 每条边的金额=TxCount笔数×平均金额，聚合金额=所有原始行金额之和
6. **筛选一致性**: 筛选条件变化后，流向图、统计表、明细表同步变化且结果一致
7. **大数据量**: 验证百万级、千万级、上亿级数据下的性能、内存、分页、懒加载、聚合、并发和索引策略

### 1.2 测试原则

- **每条可追溯**: 图上的每个数据点都可追溯到原始数据行
- **每条可验证**: 所有统计金额可通过 SQL 查询逐条验证
- **交叉验证**: 前端展示值、后端 API 返回值、数据库原始值三者一致
- **边界覆盖**: 空值、脏数据、重复、冲正、退款等异常场景全覆盖

### 1.3 强制追溯闭环

资金流向图任何测试均必须产出一份“追溯账本”，至少包含以下列，不能只看 UI 数字是否接近：

| 字段 | 说明 | 生成方式 |
|------|------|----------|
| `source_type` | 数据来源类型：`csv`、`db_import`、`manual_upload`、`synthetic` | 测试场景固定值 |
| `source_path` | CSV 绝对路径或 PG 表名 | CSV 使用原始绝对路径；PG 使用 `mz.ls_0709.交易明细信息` |
| `source_row_no` | 原始行号 | CSV 从 2 开始；PG 使用 `row_number() over (...)` 或主键/流水号 |
| `row_hash` | 原始行哈希 | 对交易时间、金额、方向、本方、对手、流水号做 SHA256 |
| `transaction_id` | 交易流水号或复合键 | 优先 `交易流水号`，缺失时使用复合键 |
| `raw_direction` | 原始收付标志 | 原始值，不允许被覆盖 |
| `normalized_direction` | 归一化方向 | `进`、`出` 或空/未知 |
| `raw_amount` | 原始金额文本/数值 | 保留小数原文用于核对 |
| `normalized_amount` | 归一化金额 | 后端参与计算的金额 |
| `edge_source` | 建图后的源节点 | 按方向规则生成 |
| `edge_target` | 建图后的目标节点 | 按方向规则生成 |
| `edge_id` | 建图边 ID | `source -> target` 或后端返回 ID |
| `included_by_filter` | 是否被当前筛选命中 | 每套筛选条件独立记录 |

通过标准：任意一条边的 `tx_count` 必须等于该边追溯账本命中行数；任意一条边的 `amount` 必须等于该边追溯账本 `normalized_amount` 逐行求和，允许误差仅为货币四舍五入后的 `0.005` 元以内。任意一个节点的 `amount_in`/`amount_out` 必须分别等于所有入边/出边追溯账本金额合计。

### 4.3 SQL 校验查询模板

所有测试用例使用以下模板进行数据核对:

```sql
-- 1. 按方向分组统计
SELECT 收付标志, COUNT(*), SUM(交易金额) FROM ls_0709.交易明细信息 GROUP BY 收付标志;

-- 2. 按主体统计流入/流出
SELECT 交易方户名, 交易账号,
  SUM(CASE WHEN 收付标志='进' THEN 交易金额 ELSE 0 END) AS total_in,
  SUM(CASE WHEN 收付标志='出' THEN 交易金额 ELSE 0 END) AS total_out,
  COUNT(*) AS txn_cnt
FROM ls_0709.交易明细信息
GROUP BY 交易方户名, 交易账号;

-- 3. 按关系统计 (本方→对手)
SELECT 
  CASE WHEN 收付标志='出' THEN 交易方户名 ELSE 对手户名 END AS source,
  CASE WHEN 收付标志='出' THEN 对手户名 ELSE 交易方户名 END AS target,
  COUNT(*) AS txn_cnt,
  SUM(交易金额) AS total_amount
FROM ls_0709.交易明细信息
WHERE 收付标志 IN ('进','出')
GROUP BY source, target;

-- 4. 按时间范围筛选
SELECT COUNT(*), SUM(交易金额) FROM ls_0709.交易明细信息
WHERE 交易时间 >= '2024-01-01' AND 交易时间 < '2024-02-01';
```

### 7.2 数据准确性验收

**硬性要求**: 以下条件必须全部满足：

1. 图上的每个金额都能追溯到原始数据行
2. 汇总统计的每一分钱都能用 SQL 验证

## 8. 缺陷判定标准

### 8.1 以下情况判定为 P0 缺陷（立即停止发布）

- 图上金额与数据库原始金额差异 ≥ 0.01 元
- 方向归一化错误（如"进"被映射为"出"）
- 某笔交易在图上完全不可见且无法通过任何筛选找到
- 节点流入/流出金额与实际交易金额不符
- 边详情中的任意一行交易金额与该行原始数据不符
- 筛选后统计结果与 SQL 差异 ≥ 1 行或 ≥ 0.01 元
- 关键 API 返回 500 错误

### 8.2 以下情况判定为 P1 缺陷

- 筛选后建图边数多于无筛选（违反单调性）
- 主体详情中身份信息缺失
- 边详情为空（应为非空时）
- 数据穿透 +/- 按钮无响应
- 大数据量下 API 超时（> 30s）
- 标签遮罩逻辑错误

### 8.3 以下情况判定为 P2 缺陷

- 导出图片不全
- 筛选面板布局异常
- 日期选择框英文
- 节点位置重置

### 8.4 以下情况不视为缺陷

- 600 条边的截断限制（这是设计）
- 空方向行不建图（这是设计）
- 失败交易也建图（当前设计如此）
- 最大 5000 条的审计限制（这是设计）

### 9.3 关键 SQL 查询汇总

```sql
-- 验证方向分布
SELECT 收付标志, COUNT(*), SUM(交易金额) FROM ls_0709.交易明细信息 GROUP BY 收付标志 ORDER BY COUNT(*) DESC;

-- 验证单账号统计
SELECT 交易账号, 交易方户名, 
  COUNT(*) as txn_cnt,
  SUM(CASE WHEN 收付标志='进' THEN 交易金额 ELSE 0 END) as total_in,
  SUM(CASE WHEN 收付标志='出' THEN 交易金额 ELSE 0 END) as total_out
FROM ls_0709.交易明细信息
GROUP BY 交易账号, 交易方户名
ORDER BY txn_cnt DESC;

-- 验证关系聚合 (用于核对 BuildFlowGraph 的边)
SELECT 
  CASE WHEN 收付标志='出' THEN 交易账号 ELSE COALESCE(NULLIF(交易对手账卡号,''), 对手户名) END as source,
  CASE WHEN 收付标志='出' THEN COALESCE(NULLIF(交易对手账卡号,''), 对手户名) ELSE 交易账号 END as target,
  COUNT(*) as edge_tx_count,
  SUM(交易金额) as edge_amount,
  MIN(交易时间) as first_time,
  MAX(交易时间) as last_time
FROM ls_0709.交易明细信息
WHERE 收付标志 IN ('进','出')
GROUP BY source, target
HAVING COUNT(*) > 0
ORDER BY edge_amount DESC;

-- 验证日期范围筛选
SELECT COUNT(*), SUM(交易金额) FROM ls_0709.交易明细信息
WHERE 交易时间 >= '2024-01-01' AND 交易时间 < '2024-02-01';

-- 验证组合筛选
SELECT COUNT(*) FROM ls_0709.交易明细信息
WHERE 交易账号 = '79040066601200056144'
  AND 收付标志 = '出'
  AND 交易时间 >= '2024-01-01' AND 交易时间 < '2024-02-01';
```
